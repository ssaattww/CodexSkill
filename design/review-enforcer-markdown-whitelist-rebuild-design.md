# review-enforcer: Markdown 許可一覧再構築の設計

## 目的

文書検査の許可一覧は、単に lint を通すための単語表ではない。リポジトリ内で「この語を文章中で使ってよい」と認める用語集であり、各語の意味を人が確認できる必要がある。

この設計では、既存文書から語彙を抽出し、頻度や表記揺れを見ながら、利用者が確認できる単位で `tools/lint/markdown-whitelist.yaml` を再構築する。

## 全体の流れ

1. 既存文書を走査する。
2. 英字語、片仮名語、漢字を含む日本語語彙を抽出する。
3. 頻度、出現元、読み、正規形、品詞、候補グループを確認する。
4. 同じ概念に見える語を小さな候補グループに分ける。
5. 利用者が canonical 表記、許可する別表記、修正すべき揺れ表記を判断する。
6. 承認された語だけを `tools/lint/markdown-whitelist.yaml` に追加する。
7. 修正すべき揺れ表記は `tools/lint/prh.yml` に追加する。

## 利用者が判断すること

利用者は、抽出された全語を一括承認しない。確認する単位は、意味が近い小さな候補グループである。

各候補グループについて、利用者は次を決める。

- 正式に使う表記はどれか。
- 同じ意味の別表記として残してよいものはどれか。
- 今後は正式表記へ直すべき表記はどれか。
- 許可一覧に入れる場合、その語の説明は何か。
- 実際に追加、変更、削除される `term`、`aliases`、`description` を承認するか。

lint 結果、頻度の高さ、エージェントのレビュー結果は、利用者の承認の代わりにはならない。

## エージェントが行うこと

エージェントは、候補抽出と許可一覧編集を分けて扱う。

エージェントはまず語彙抽出を実行し、候補を提示する。次に、利用者が確認しやすいように、頻度、出現元、読み、正規形、品詞、候補グループをまとめる。

エージェントは、利用者が明示確認する前に大量の許可語を追加してはならない。許可一覧を編集する場合は、承認済みの小さな候補グループだけを反映する。

## whitelist と prh の使い分け

`tools/lint/markdown-whitelist.yaml` は、文章中で使ってよい語を定義する。

入れるべきものは次である。

- 正式な製品名、機能名、プロジェクト固有語。
- 文書中に残す必要がある専門語。
- 長期的に許可する別表記。

`tools/lint/prh.yml` は、表記揺れを検出して正式表記へ寄せるために使う。

入れるべきものは次である。

- 誤表記。
- 古い名称。
- 今後は正式表記へ直すべき表記。
- 文章中に残したくない英字表記や片仮名表記。

つまり、許可したい別名は `aliases`、直したい揺れは `prh` に入れる。

## 許可一覧 entry の形

許可一覧の各 entry は、少なくとも `term` と `description` を持つ。

```yaml
entries:
  - term: Tracker.DebugHost
    aliases:
      - DebugHost
    description: 診断用実行体の名称。
```

`term` は許可する語である。`description` は、その語をなぜ許可するのか、人が意味を確認するための説明である。`aliases` は、同じ意味として今後も許可する別表記だけに使う。

複合語やフレーズを優先する。単一の汎用語を登録すると、意図しない文章まで通るため、最終手段とする。

`description` 内の語も lint 対象である。そのため、説明文に新しい英字語、片仮名語、日本語語彙を入れる場合も、同じ方針で扱う。

## 抽出結果の読み方

SudachiPy 版の抽出結果には、主に次の情報が出る。

- `surface`: 文書に出てきた表記。
- `normalized`: 正規化された表記。
- `reading`: 読み。
- `partOfSpeech`: 品詞。
- `groupKey`: 近い候補を並べるためのキー。
- `count`: 出現回数。
- `sources`: 出現元。

`groupKey`、読み、正規形は候補を探すための手がかりであり、同じ意味であることを自動確定するものではない。たとえば `namespace`、`ネームスペース`、`名前空間` の対応は、最終的に利用者が判断する。

## 検査対象外にするもの

次は文章ではないため、検査対象から外す。

- コードフェンス。
- インラインコード。
- URL や link destination。
- Markdown のリンクアドレス部。
- 脚注定義行。

リンクの表示テキストは文章なので検査対象である。

## 完了条件

許可一覧再構築は、次を満たしたときだけ完了とする。

- 語彙抽出結果から候補グループが作られている。
- 利用者が追加、変更、削除される entry を明示確認している。
- `whitelist` と `prh` の使い分けが説明できる。
- 大量自動追加で lint を通していない。
- 意図的に失敗する lint gate がある場合、その理由が報告や進捗管理に記録されている。
