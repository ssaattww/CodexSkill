# 責務配置ポリシー

local skill の新規作成や実質更新で、判断責務を上位 skill に寄せるか、下位 skill に残すかを決めるときは、この基準を使う。

## 基本原則

- 判断責務は、できるだけその判断を最終利用する親 skill に置く。
- ただし、`1対多` の呼び出し関係で親側未決定のまま下位 skill が単独利用されうる場合は、安易に上位へ寄せすぎない。
- 下位 skill は、親が決めるべきことを再判断してはならない。
- 下位 skill は、親判断が前提であることを明示しつつ、単独で読んでも破綻しない最小限の前提だけを持つ。

## 上位に寄せてよい条件

次をすべて満たすときだけ、判断責務を上位 skill に寄せてよい。

1. 呼び出し元が実質固定されている。
2. その判断が、その親子サブツリーの内部で閉じている。
3. 下位 skill を単独利用しても、その判断欠落で運用が破綻しない。
4. 同じ判断を複数の sibling で重複させるより、上位一箇所に置いた方が drift を防げる。

## 上位に寄せすぎてはいけない条件

次のどれかに当てはまるときは、判断責務を上位へ寄せすぎない。

- 親から複数の下位 skill に分岐する `1対多` 構造である。
- 下位 skill が他の入口からも利用されうる。
- 親がその判断を通らなくても下位 skill が直接使われる可能性が高い。
- 上位にだけ判断を書くと、下位 skill 単体の読み手が前提を見失う。

この場合は、次の形にする。

- 判断 owner は上位 skill に置く。
- 下位 skill には「誰の判断に従うか」を書く。
- 下位 skill には必要最小限の参照だけ残し、独自の既定方針は作らない。

## skill 修正時の扱い

- 既存 skill の修正でも、この基準を新規作成時と同じように使う。
- 既存の責務分担に重複や逆転が見つかったら、修正時に解消を検討する。
- ただし、呼び出し元が固定されていない skill 群で、無理に責務を上位へ集約しない。

## 具体例

### 上位に寄せやすい例

- `development-orchestrator -> skill-authoring-wrapper`
  - caller が実質固定されやすく、local skill authoring の入口を一本化しやすい。
- `git-workflow-manager -> git-commit-manager`
  - Git 提出フロー内部で閉じる判断なら、commit 形の既定方針を上位で持ちやすい。

### 上位に寄せすぎない例

- `codex-delegation-executor -> implementation-executor / design-executor / tdd-executor`
  - `1対多` で分岐しやすく、下位 skill を単独参照する場面もあるため、下位は上位の決定に従うことを明記しつつ、独自判断 owner にはしない。
