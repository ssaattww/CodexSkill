# 開発上の問題一覧

`/home/ibis/dotnet_ws/ExcelReport/tasks/feedback-points.md` をもとに、再利用可能な「開発 process 上の問題」を抽出した一覧。

issue 固有の仕様決定や機能設計の詳細は除外し、他リポジトリでも再発しうる問題だけを残す。

## 1. 委譲不足

問題:

- 調査、レビュー、ビルド、テスト、確認作業を人手で行ってしまい、Codex への委譲が徹底されない
- 実行結果が `reports/` に証跡として残らない

代表 FP:

- FP14
- FP20
- FP23
- FP25
- FP146

## 2. タスク分解不足

問題:

- Codex への依頼が大きすぎて曖昧になる
- 一度に多くの作業を抱え、段階的な前進にならない

代表 FP:

- FP22
- FP31
- FP43

## 3. 進捗同期不足

問題:

- `tasks-status.md` / `phases-status.md` の更新が後回しになる
- 現在地、残件、完了条件が記録に即時反映されない

代表 FP:

- FP16
- FP142
- FP143
- FP151

## 4. TDD と検証不足

問題:

- 実装がテスト先行にならない
- 実行結果ではなく口頭報告で済ませてしまう
- unit だけで終わり、integration / E2E で契約を確認しない

代表 FP:

- FP17
- FP26
- FP42
- FP59

## 5. レビュー運用不足

問題:

- レビューなしで完了扱いしてしまう
- 複数タスクをまとめてレビューして粒度が崩れる
- レビュー結果の記録が弱い

代表 FP:

- FP30
- FP33
- FP34
- FP90
- FP91
- FP152

## 6. コストと実行安定性の管理不足

問題:

- 無駄な再実行、過剰な並列実行、重い依頼の投げ方で Codex 実行コストが増える
- reasoning effort の選択が作業内容に対して不適切になる
- 使用量上限に達したときの扱いが曖昧

代表 FP:

- FP18
- FP31
- FP32
- FP43

## 7. Git / PR ワークフロー不足

問題:

- ブランチ分離、PR 提出、レビュー可能な単位での履歴管理が徹底されない
- コマンド実行パターンが実行環境の許可制約に合わない

代表 FP:

- FP15
- FP46
- FP99
- FP102

## 8. issue 要件確定不足

問題:

- issue 本文を読まずにローカル推測で進めてしまう
- `gh` が使えないときの代替取得経路が定義されていない

代表 FP:

- FP74
- FP111

## 9. 自走と停止条件の境界不足

問題:

- 不要な確認待ちで止まりすぎる
- 逆に、止まるべき場面の境界が曖昧なまま進みやすい

代表 FP:

- FP21
- FP141
- FP144

問題:

- 実装が設計更新より先に進みやすい
- 破壊的変更の記録、予定版数、確定版数の扱いが曖昧になる

代表 FP:

- FP39
- FP40
- FP75
- FP76
- FP77
- FP110

## 11. コーディング規約の強制不足

問題:

- public/protected API の XML doc など、反復して要求される規約がレビュー前に機械的に確認されない

代表 FP:

- FP19

## 12. 命名と記録品質不足

問題:

- issue 番号だけで意味が分からない命名をしてしまう
- 読み手が内容を把握しづらい記録が残る

代表 FP:

- FP24
- FP147

## 除外対象

以下はこの一覧には含めない。

- DSL 仕様や Warning 方針などの feature 固有仕様
- issue #58 / #61 など特定 issue の設計詳細
- release 自動化そのものの変更要求

これらは feedback の process 問題ではなく、issue / task / design / operations の個別管理対象とする。
